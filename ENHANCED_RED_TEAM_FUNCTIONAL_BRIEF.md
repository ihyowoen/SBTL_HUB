# Enhanced Red-Team Functional Brief

## 목적
이 문서는 `SBTL_HUB`의 다음 작업에서 **무엇을 만들고**, **무엇을 만들면 안 되는지**, 그리고 **완료 기준이 무엇인지**를 명확히 하기 위한 기능 명세 + red-team 점검 메모이다.

빌드 성공만으로는 완료가 아니다.  
완료 기준은 **실제 배포본에서 사용자가 체감하는 핵심 흐름이 정상 작동하는지**이다.

---

## 1. 핵심 방향 정리

### 1) 번역 기능에 대한 정확한 요구사항
요구사항은 **별도의 번역기 탭/제품**을 만드는 것이 아니다.

원하는 것은 다음이다.
- 외국 뉴스/카드가 허브에 올라왔을 때
- 기존 `NEWS` / `AI` 흐름 안에서
- 사용자가 해당 기사에 대해
  - 번역
  - 한국어 요약
  - 핵심 설명
  - 임플리케이션 정리
  를 받을 수 있어야 함.

즉, 번역은 **독립 툴**이 아니라 **기존 뉴스 읽기 경험을 강화하는 보조 기능**이어야 한다.

### 2) 웹툰 기능에 대한 정확한 요구사항
웹툰은 단순히 카드 디자인이 아니라, **시리즈 단위로 끊김 없이 이어지는 읽기 경험**이 중요하다.

필수 조건:
- 허브 내 웹툰 카드에서 각 에피소드가 직접 열릴 것
- EP.1에서 EP.2로 자연스럽게 넘어갈 것
- Bonus 편도 정상 열릴 것
- 시리즈 랜딩 페이지와 실제 HTML 파일이 서로 같은 경로를 가리킬 것

### 3) AI 봇에 대한 정확한 요구사항
현재 AI 봇은 단순 FAQ/텍스트 출력 수준에서 더 고도화되어야 한다.

필수 방향:
- 카드 검색 결과를 클릭 가능한 링크로 보여줄 것
- 외부 검색 결과도 제목/요약/링크 구조로 보여줄 것
- 질문 유형에 따라 답변 포맷을 바꿀 것
- 1차 퀵가이드만이 아니라 2차/3차 퀵가이드까지 이어질 것

---

## 2. 절대 만들면 안 되는 것

다음은 **잘못된 방향**이다.

### 금지 1) 별도 Translation / Translator 탭 추가
잘못된 정보구조다.
사용자 니즈는 “번역기 하나 만들기”가 아니라,  
**허브에 이미 떠 있는 외국 뉴스/카드를 이해하기 쉽게 만드는 것**이다.

### 금지 2) 빌드 성공만을 완료로 간주
`npm run build` 또는 Vercel 배포 성공은 **최소 조건**일 뿐이다.
실제 클릭/이동/링크/렌더가 안 되면 미완료다.

### 금지 3) 링크를 문자열로만 출력
`https://...` 텍스트를 채팅 버블에 보여주는 것만으로는 부족하다.
링크는 반드시 **실제 클릭 가능한 anchor**로 렌더되어야 한다.

### 금지 4) 문맥 없는 퀵가이드 버튼 남발
버튼 수를 늘리는 것이 목적이 아니다.
2차/3차 퀵가이드는 **이전 질문/답변 문맥에 따라 달라져야** 한다.

## 3. 기능 우선순위

### 우선순위 1 — 웹툰 즉시 복구
`src/App.jsx`의 `WEBTOON_COLLECTIONS.items`에 `url`이 반드시 있어야 한다.

예시:

```js
items:[
  {
    id:101,
    title:"EP.01 LFP의 두 가지 착각",
    subtitle:"좋은 배터리일 수 있다. 그런데 너무 쉽게 보면 핵심을 놓친다.",
    date:"2026.04.05",
    isNew:true,
    color:"#F85149",
    likes:0,
    url:"/webtoon/lfp-two-misconceptions-ep1.html",
    tag:"START"
  },
  {
    id:102,
    title:"EP.02 덜 타 보인다고, 덜 위험한 건 아니다",
    subtitle:"보이는 화재 위험과 보이지 않는 off-gas 위험은 같은 말이 아니다.",
    date:"2026.04.05",
    isNew:true,
    color:"#D29922",
    likes:0,
    url:"/webtoon/lfp-two-misconceptions-ep2.html",
    tag:"PART 2"
  },
  {
    id:103,
    title:"BONUS.01 누가 진짜 돈을 내나?",
    subtitle:"끝날 때 비용은 존재한다. 그 돈을 누가 떠안는지 묻는 보너스 에피소드.",
    date:"2026.04.05",
    isNew:true,
    color:"#58A6FF",
    likes:0,
    url:"/webtoon/who-pays-bonus1.html",
    tag:"BONUS"
  }

```

추가 확인 필수:
- `public/webtoon/lfp-two-misconceptions-ep2.html`
- `public/webtoon/who-pays-bonus1.html`
- `public/webtoon/lfp-two-misconceptions-ep1.html` 내부의 “다음 화 보기” 링크

핵심은 **App 쪽 데이터 + 실제 HTML 파일 + 화살표 링크**를 모두 맞추는 것이다.

---

### 우선순위 2 — 챗봇 링크 클릭 가능 구조
현재처럼 URL을 문자열 텍스트로만 출력하면 안 된다.

권장 구조:

```js
{
  role: "assistant",
  content: "관련 카드 3개를 찾았어.",
  links: cards.map(c => ({
    title: c.T,
    url: c.url,
    source: c.src,
    date: c.d
  }))
}
```

렌더 예시:

```jsx
{m.links?.map(link => (
  <a
    key={link.url}
    href={link.url}
    target="_blank"
    rel="noopener noreferrer"
    style={{display:"block", marginTop:8, color:t.cyan, textDecoration:"none"}}
  >
    {link.title} ↗
  </a>
))}
```

필수 변경점:
- 카드 검색 결과 → 클릭 가능한 카드 리스트
- 외부 검색 결과 → 제목 + 짧은 요약 + 링크 목록
- 원문 제목/원문 링크/날짜/출처 정보를 유지

---

### 우선순위 3 — AI 봇 고도화
현재 AI 봇은 얕다. 다음 수준까지는 가야 한다.

#### 질문 유형 분기
최소한 다음 4종으로 나눠야 한다.
- 요약형
- 비교형
- 정책 일정형
- 최근 뉴스형

#### 각 유형별 권장 출력
- **요약형**: 핵심 3줄 + 관련 카드 링크
- **비교형**: 차이점 3개 + 관련 기사/카드
- **정책 일정형**: 다가오는 일정 + 지역 + 업데이트 날짜
- **최근 뉴스형**: 최신 카드 + 외부 기사 링크 3개 + 후속 질문

---

### 우선순위 4 — 1차/2차/3차 퀵가이드
현재 1차 퀵가이드만 있는 것은 부족하다.

#### 1차 퀵가이드 예시
- 오늘 핵심 뉴스 3개
- LFP 리스크 한 번에
- 한국 정책 일정 뭐 있어?
- 먼저 봐야 할 카드 추천

#### 2차 퀵가이드 예시
- 조금 더 쉽게 설명해줘
- 관련 카드 더 보여줘
- 최신 기사 링크로 보여줘
- 정책 일정만 따로 정리해줘

#### 3차 퀵가이드 예시
**뉴스형 답변 후**
- 오늘 기준으로 다시
- 미국만 따로
- 중국만 따로
- 링크 3개만 보여줘

**정책형 답변 후**
- 다가오는 일정만
- 한국/EU 비교
- 실무 영향만 요약

**기술형 답변 후**
- 쉽게 비유해서 설명
- LFP랑 비교
- 왜 중요한지 한 줄로

중요:
퀵가이드는 **많이 보여주는 것**이 아니라 **문맥에 맞게 이어주는 것**이 목적이다.

---

## 4. Red-Team 관점에서 추가 보완할 점

### 1) 사용자가 잘못 생각할 수 있는 부분
“번역 기능”을 넣자고 할 때, 쉽게 **범용 번역기 제품** 쪽으로 흐를 수 있다.  
그건 스코프가 틀린다.

정확한 목표는:
- 외국 뉴스를
- 허브 문맥 안에서
- 더 잘 이해하고 활용하게 만드는 것
이다.

### 2) 추적 가능성 유지
번역/요약이 들어가면 원문 정보가 사라질 위험이 있다.
이건 신뢰도를 해친다.

필수 유지 항목:
- 원문 제목
- 원문 링크
- 날짜
- 출처
- 번역/요약임을 알리는 표시

### 3) AI 봇이 “똑똑한 검색창” 정도로만 보이면 안 됨
현재 수준에서 개선해야 할 핵심은 단순 대답이 아니라 **후속 탐색 경험**이다.
즉,
- 답을 주고 끝나는 봇 ❌
- 답 → 링크 → 후속 질문으로 이어지는 봇 ✅

### 4) 웹툰은 카드만 열리면 끝이 아님
사용자는 “시리즈를 이어서 읽는 경험”을 기대한다.
따라서 각 화 간 내비게이션과 시리즈 랜딩 페이지 정합성이 중요하다.

---

## 5. 완료 기준 (Acceptance Criteria)

다음이 모두 충족되어야 완료로 본다.

### 웹툰
- TOON에서 EP.2가 직접 열린다
- EP.1 → EP.2 이동이 된다
- BONUS 페이지가 열린다
- 시리즈 랜딩 페이지의 링크와 실제 HTML 경로가 일치한다

### 챗봇
- 카드 검색 결과 링크가 클릭된다
- 외부 기사 링크가 클릭된다
- 외국 기사에 대해 번역/한국어 요약/핵심 설명이 기존 AI 흐름 안에서 가능하다
- 2차/3차 퀵가이드가 문맥적으로 이어진다

### 정보 구조
- Translation / Translator 탭이 새로 생기지 않는다
- 기존 상단 탭 구조는 단순하게 유지된다

---

## 6. 구현 후 보고 형식
작업 완료 후에는 아래 형식으로 짧게 정리해야 한다.

1. 수정한 파일 목록
2. 각 파일에서 무엇을 고쳤는지
3. 실제 배포본에서 어떤 동작을 확인했는지

예시:
- `src/App.jsx`: 웹툰 url 추가, 챗봇 links[] 렌더, 2차/3차 퀵가이드 추가
- `public/webtoon/lfp-two-misconceptions-ep2.html`: 본문 복구 및 내비 링크 수정
- `public/webtoon/who-pays-bonus1.html`: 본문 복구

동작 확인:
- EP.2 직접 열림
- EP.1 → EP.2 이동 확인
- AI 봇 카드 링크 클릭 확인
- 외국 기사 번역/요약 버튼 동작 확인

---

## 결론
이 작업의 목표는 **새 기능을 마구 늘리는 것**이 아니다.

정확한 목표는:
- 웹툰 시리즈 흐름을 끊김 없이 복구하고
- AI 봇을 단순 FAQ가 아니라 뉴스 이해/탐색 도구로 끌어올리며
- 외국 뉴스 번역을 기존 읽기 경험 안에 자연스럽게 통합하는 것
이다.

빌드 성공은 최소 조건일 뿐이다.  
사용자 기준에서 실제로 눌리고, 이어지고, 이해되는 경험이 만들어져야 완료다.
