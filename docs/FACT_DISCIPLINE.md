# FACT DISCIPLINE — Locked

## 0. 이 문서의 역할

SBTL_HUB는 산업 인텔리전스 뉴스레터다. 뉴스는 **정확성**이 생명이고, 잘못된 정보를 전달하면 프로덕트 신뢰가 끝난다.

이 문서는 **카드에 들어가는 모든 fact·숫자·인용**에 대한 절대 기준을 잠근다. 다른 모든 문서의 규칙보다 상위에 있으며, 충돌 시 본 문서가 우선한다.

**User 명시 입장 (2026-04-17):** *"뉴스라서 정확성이랑 데이터가 생명이야. 잘못된 정보를 전달하면 끝나. 절대금지절대."*

---

## 1. 절대 금지 — PROHIBITED

다음은 **예외 없이 금지**된다. 각 항목은 실제로 발생했던 위반 패턴을 반영한 것이다.

### 1.1 기억·훈련 데이터에서 가져온 숫자·패트 인용
- 금지 예: "삼성SDI 전고체 양산 2027년, SK온 2028년, LG엔솔 2030년" — 출처 fetch 없이 기억에서 기재
- 금지 예: "CATL 시장 점유율 37%" — 원문 미확인 통계
- 금지 예: "전고체 에너지밀도 500Wh/kg" — 일반 상식으로 네승
- 기억이 맞다 하더라도 **원문 fetch 없이는 fact 필드에 쓰지 않는다**

### 1.2 출처 없는 숫자 생성 (hallucinated 수치)
- 금지 예: "한국 운전자가 연간 수십만원 수익" — 근거 없이 추론으로 생성된 정량
- 금지 예: "월 평균 충전 비용 3만원" — 보도에 없는 판단 수치
- 클레어가 그럴듯한 수치를 요청해도, 근거 없으면 **작성하지 않고 비워둔다**

### 1.3 비교 기준 없는 "과거 대비" 서술
- 금지 예: "과거 일반적이던 5~10년에서 3~5년으로 압축" — "과거" 출처 없음
- 금지 예: "이전 가이덗 수정" — "이전" 가이드 문서 미확인
- 비교는 **두 시점 모두 출처 확보**했을 때만 허용

### 1.4 환산 근거 없는 통화 환산
- 금지 예: "300억 위안(약 5.7조원)" — 환율 기준 명시 없음
- 허용 예: "300억 위안(약 5.7조원, 2026-04-16 종가 환율 1위안=190원 기준)"
- 환율은 **원문에 명시된 수치 또는 인용 당일 종가** 중 하나만

### 1.5 기억 기반 기업 로드맵·타임라인 인용
- 금지 예: "가이던 로드맵은 X이었으나 수정" — 원문 fetch 없이 기억에서 쓴 기업 계획
- 기업 계획은 수시로 변경되므로 **최신 IR · 공시 · 보도자료 원문** 에만 근거

### 1.6 추론을 fact와 같은 어보로 서술
- 금지 예: "K-업체는 이에 대응해 양산 일정을 당긴다" — 추론인데 fact로 보임
- 금지 예: "2Q26 IR에서 메시지가 나온다" — 예측인데 단정형
- 추론은 implication 필드에서 **관찰형 어휘**로만 가능 (§4 참조)

## 2. 절대 의무 — REQUIRED

### 2.1 fact 작성 전 web_fetch 의무
- 모든 passed_spec의 `primary_url`은 Prompt B 실행 시 실제 web_fetch 후 본문에 기재
- fetch 실패 시에는:
  - 다른 URL (동일 이벤트의 다른 매체) fetch 시도
  - 모두 실패 시 **카드 생성 중단** 또는 `_fact_unverified=true` 플래그 + `_needs_review=true`

### 2.2 fact_sources 필드 기록
각 카드에 다음 필드를 **의무적으로** 추가한다:

```json
"fact_sources": [
  {
    "claim": "2GWh 전고체 공장 투산",
    "source_url": "http://www.cbea.com/djgc/202604/963162.html",
    "source_quote": "恩力动力安徽2GWh固态电池先进智造基地正式投产",
    "fetched_at": "2026-04-17T04:30:00"
  },
  ...
]
```

- fact 필드에 등장하는 **모든 숫자·고유명사·인용·날짜**는 `fact_sources`에 대응하는 `claim`이 있어야 한다
- 대응 없으면 해당 수치는 fact에서 **제거**

### 2.3 불확실 시 비워두기
- 활력적인 카드의 대부분 필드에 정보가 있으면 OK
- 일부 필드 정보 부족 → 그 필드가 짧아지거나 implication이 적어지는 것은 정당
- **빈 칸을 기억·추론으로 메우는 것이 잘못된 정보의 주 원인**

### 2.4 Prompt C red-team 단계 fact 검증
- 각 카드 fact의 모든 숫자·고유명사·인용·날짜가 `fact_sources`에 대응하는지 자동 스캔
- 대응 없는 수치·인용 검출 시 카드 **payload에서 제외** (silent discard 아니라 명시 제외)
- 제외된 카드는 `W7Rx_redteam_report.md`에 사유 기록

## 3. fact 필드 — 허용 / 금지

### 3.1 허용되는 것
- 원문에 **명시적으로** 적힌 수치·날짜·주체·인용
- 원문 문장을 한국어로 번역·요약 (의미 기벘 나가지 않는 접속사·표현 변경)
- "X에 따르면" 형태의 attribution
- 동시에 보도된 복수 출처의 교집합 정보

### 3.2 금지되는 것
- 원문에 없는 숫자 추가 (기억·추론·상식무관)
- 원문에 없는 비교·해석·맥락 (그것은 implication 영역)
- 기억에서 가져온 로드맵·타임라인
- "업계에서는" 같은 출처 불분명 단체 인용
- "시장에서는", "분석가들은" 같은 못 검증 판단의 모호한 주체

---

## 4. implication 필드 — 단정형 금지 / 관찰형 허용

implication은 "fact의 결과"가 아니라 "편집자의 관찰·해석"이다. 어휘가 잠재적 엄격성을 드러내야 한다.

### 4.1 금지 — 단정형
- "X가 나온다" / "X한다"
- "X 가능성이 증가"
- "X에 반영된다"

### 4.2 허용 — 관찰형
- "X할 여지가 있다" / "X하는 방향으로 열린다"
- "X 신호가 보인다"
- "X가 관찰 포인트"
- "X인지 확인이 필요"
- "X가 검토되는 구간에 진입"

### 4.3 implication에도 적용되는 정확성 원칙
- implication이 지칭하는 주체(K 3사, 현대차, CATL 등)는 일욥지만 사실이어야 한다
- "삼성SDI 2027 양산" 같은 세부 수치는 implication에도 쓸 수 없음 — fact 기준 동일
- 사실 기반이 아닌 implication은 아무 의미 없는 문장 (§1.5 윈반)

---

## 5. 숫자 인용 규칙

### 5.1 원문 활력적 숫치에만 근거
- 원문 "1 GWh" → fact에 "1GWh"
- 원문 "300억 위안" → fact에 "300억 위안" 또는 "300억 위안(약 44억 달러, 보도 시점 환율 기준)"
- 원문에 없는 보조 수치 추가 금지

### 5.2 환산 규칙
- 원문에 환산이 있으면 그대로 인용
- 원문에 환산이 없으면 추가 하지 않거나, 추가할 경우 **환율 출처 + 기준 날짜** 반드시 명시
- 예: "300억 위안(보도 시점 당일 기준 약 44억 달러)" — 허용
- 예: "300억 위안(약 5.7조원)" — 금지 (환율 기준 없음)

### 5.3 근사치 표기
- 원문 "about 2 trillion yuan" → "약 2조 위안" (OK)
- 원문 "2 trillion yuan" → "2조 위안" (OK, 근사치 임의 금지)
- 수치 정밀도를 원문보다 높이면 안 됨

## 6. fact_sources 필드 스키마

```json
"fact_sources": [
  {
    "claim": "string — fact에 등장한 주장 (숫자·고유명사 단위)",
    "source_url": "string — 작에 fetch된 URL",
    "source_quote": "string — 원문에서 직접 인용한 구절/문장 (30자 내외)",
    "fetched_at": "string — ISO 8601"
  }
]
```

### 6.1 입력 규칙
- 각 `claim`은 fact 본문에서 검색 가능해야 함 (키워드 기반)
- `source_quote`는 원문에서 **직접 복사·붙여넣기 가능한 단어·문장**. 자체 번역 불가
- `fetched_at`은 `web_fetch` 시점

### 6.2 총계
- `fact_sources`의 URL은 `urls` 배열에도 존재해야 함 (불일치 시 스카웃)
- 빈 배열 또는 null → 해당 카드는 payload에 포함 불가

## 7. Pre-publish Self-check

Prompt C 그림자가 각 카드를 payload에 포함시키기 전 다음을 진행한다:

### 7.1 fact 검증
- [ ] fact에 등장하는 모든 숫자·날짜·고유명사·인용이 `fact_sources`의 `source_quote`에서 검색됨
- [ ] fact에 "업계" / "시장은" 같은 모호 주체 없음
- [ ] fact에 환산 수치 있다면 환율 출처 명시됨
- [ ] fact에 "이전·과거·다른 사례" 비교가 있다면 그 비교 대상도 `fact_sources`에 포함됨

### 7.2 implication 검증
- [ ] 모든 implication 항목이 관찰형 어휘로 끌될지 / 단정형 사실 주장 없음
- [ ] implication 안에 등장하는 수치도 `fact_sources`에 대응 있음 (implication은 "fact의 확장"이지 "새 fact 생성"이 아닐)
- [ ] K-기업 로드맵·타임라인이 전제로 쓰였다면 그 전제도 소스 있음

### 7.3 시스템 체크
- [ ] `fact_sources` 내 모든 `source_url`이 `urls` 배열에 존재
- [ ] `fact_sources`가 비어있지 않음
- [ ] 모든 `fetched_at` 타임스탬프가 최근 48시간 이내

**한 항목이라도 fail → 카드 payload에서 제외**

## 8. 실제 위반 사례 — W7R3

다음은 본 문서 제정 계기가 된 실제 위반 사례다. 동일 유형을 향후 반복하지 않도록 기록한다.

### 8.1 fact 위반
사례: "삼성SDI 2027년, SK온 2028년, LG엔솔 2030년" (implication에 기재)
- 문제: K 3사 전고체 양산 타임라인은 IR에서 지속적으로 조정됨. 특히 SK온 2028·LG엔솔 2030은 출처 미확인 기억 인용
- 올바른 처리: 각 기업 IR/공시 최신본 fetch 후만 인용. fetch 불가 시 해당 숫치 제거

### 8.2 fact 위반
사례: "연간 수십만원 수익" (V2G 카드)
- 문제: 원문에 없는 정량 추정치를 fact 탕으로 서술
- 올바른 처리: "수익 모델의 구체적 산정은 서비스 출시 후 공개 예정"으로 끌내거나, 액수치 아예 언급 안 함

### 8.3 fact 위반
사례: "과거 일반적이던 5~10년에서 3~5년으로 압축" (전고체 카드)
- 문제: "과거 5~10년" 주장의 근거 없음. 비교 자체가 추쳡
- 올바른 처리: 원문에 적힌 "3~5년"만 표기. 비교하려면 이전 보도 URL도 같이 fetch 후 두 수치 모두 `fact_sources`에 등록

## 9. 최종 원칙 — 기각

1. **모르면 비워둔다.** 기억·추론·상식으로 반각자국 안 생성한다.
2. **원문에서 보지 않았으면 쓰지 않는다.** "당연해 보이는" 숫치도 검증 없이는 위반.
3. **추론과 사실을 어휘로 분리한다.** implication은 관찰형, fact는 인용형.
4. **한 카드의 실수는 전체 플랫폼의 신뢰를 긬다.** 원칙은 taste가 아니라 방화벽이다.

**정보가 없으면 카드는 생성하지 않는다.**
**정확성은 영어 번역이 아니라 생명선이다.**
